## Run against a local Jahia instance

1. Start Jahia 8.x (tested against **8.2.3**).
2. Deploy `permalink-generator` â€” install the OSGi bundle from `../target/` via the Jahia modules UI or karaf.
3. Run the tests:

```bash
# Headless
yarn e2e:ci

# Interactive (opens Cypress UI)
yarn e2e:debug
```

By default Cypress targets `http://localhost:8080` with credentials `root / root1234`. Override:

| Env variable | Default | Purpose |
|---|---|---|
| `CYPRESS_BASE_URL` | `http://localhost:8080` | Jahia base URL |
| `CYPRESS_SUPER_USER_PASSWORD` | `root1234` | Jahia admin password |

Run a single spec:

```bash
yarn e2e:ci --spec "cypress/e2e/01-auto-generation.cy.ts"
```

## Test Scenarios

Tests run in order. Each spec is numbered to express dependencies.

| Spec | Scenario | What it tests |
|---|---|---|
| `00-setup.cy.ts` | Site setup | Creates `plgentest` (module enabled) and `plgenaudit` (pages created before module enable). |
| `01-auto-generation.cy.ts` | Auto-generation | Title set â†’ vanity created. Default language has no prefix; non-default prefixed `/fr/`. |
| `02-title-rename.cy.ts` | Title rename | Page title change replaces active vanity; old URL kept as redirect. |
| `03-branch-move.cy.ts` | Branch move | Page move updates prefix, preserves slug; all descendants updated; old vanities kept as redirects. |
| `04-audit-panel.cy.ts` | Audit generation | Pages with no vanity â†’ `generatePermalinks` action creates them in bulk via API. |
| `05-force-regen.cy.ts` | Force regeneration | SMART mode skips manual vanity; `force=true` overwrites it. |
| `06-manual-vanity.cy.ts` | SMART mode | Module never overwrites manual vanities on rename; updates prefix on move. |
| `99-teardown.cy.ts` | Teardown | Deletes both test sites. |

## Architecture

### Helpers â€” `cypress/support/permalinkgen.ts`

Core library imported by all specs. Key exports:

| Export | Purpose |
|---|---|
| `SITE_KEY` | `'plgentest'` |
| `SITE_KEY_AUDIT` | `'plgenaudit'` |
| `createTestSite()` | Creates site + enables module |
| `createAuditSite()` | Creates site only (module enabled later) |
| `createPage(parent, name, titleEn, titleFr)` | Adds a `jnt:page` node |
| `publishNode(pathOrId)` | Publishes to live workspace |
| `setPageTitle(path, lang, title)` | Mutates `jcr:title` via GraphQL |
| `movePage(pathOrId, destParent)` | Moves node via GraphQL |
| `makeVanityManual(path, lang)` | Removes `jmix:permalinkGenerated` mixin via Groovy |
| `makeVanityAutoGenerated(path, lang)` | Adds `jmix:permalinkGenerated` mixin via Groovy |
| `createManualVanityUrl(path, lang, url, siteKey)` | Creates a vanity URL without the module mixin |
| `getVanityUrls(path)` | GraphQL query returning all vanity URLs for a node |
| `waitForVanityUrl(path, lang, timeout?)` | Polls until an active+default vanity exists |
| `waitForVanityUrlContaining(path, lang, substring, timeout?)` | Polls until vanity URL contains substring (use when URL is changing) |
| `assertNoVanityUrlChange(path, lang, expectedUrl)` | Asserts vanity URL has not changed |

### Groovy execution

`cy.executeGroovy(scriptFile, replacements)` from `@jahia/cypress` reads a `.groovy` file from `cypress/fixtures/`, performs string replacements, and POSTs it to the Jahia provisioning API. Placeholders are simple strings replaced literally:

```typescript
cy.executeGroovy('groovy/permalinkgen/makeVanityManual.groovy', {PATH: nodePath, LANG: 'en'})
```

### Action endpoint

The `generatePermalinks` Jahia action accepts:

```
POST /cms/render/default/en/modules/permalink-generator/templates/site-settings-base/permalinkGeneratorSettings/pagecontent/permalinkGeneratorSiteSettings.generatePermalinks.do
Content-Type: application/x-www-form-urlencoded
X-Requested-With: XMLHttpRequest

nodeIds[]=<uuid>&languages[]=en[&languages[]=fr][&force=true][&preview=true][&bypassExcluded=true]
```

Tests 04 and 05 call this directly via `cy.request()`, bypassing the admin UI (which is rendered inside a GWT iframe inaccessible to Cypress).

The module template path is used (not a site path) because the `permalinkGeneratorSiteSettings` content node under a site is only created when the settings page is first rendered in a browser. The module template node always exists.

## Test isolation

- Tests 00â€“99 run sequentially; Cypress retains cookie/session state across specs in a single run.
- Test 05 leaves `page-about` EN vanity in MANUAL state. Test 06 `before()` explicitly restores it with `makeVanityAutoGenerated()`.
- `waitForVanityUrlContaining()` must be used (not `waitForVanityUrl()`) when asserting a URL *changed*, to avoid a false positive from the old URL already satisfying the existence check.

## Writing new tests

1. Import helpers from `../support/permalinkgen`.
2. Use `waitForVanityUrlContaining` when you expect the URL to change; `waitForVanityUrl` when you only need existence.
3. If you need to call the action endpoint, use the module template path (`/cms/render/default/en/modules/permalink-generator/templates/...`) â€” site-specific settings nodes are only created when the settings page is first rendered in a browser, so they don't exist in fresh test sites.
4. Add Groovy fixtures to `cypress/fixtures/groovy/permalinkgen/` for JCR state manipulation that GraphQL mutations can't handle.
